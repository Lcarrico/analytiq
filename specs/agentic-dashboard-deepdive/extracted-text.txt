AnalytIQ
Agentic Dashboard Generation Workbench
Deep-dive implementation analysis, defect audit, and completion blueprint
Assessment date
July 6, 2026
Scope
Create Workbench: ask → plan → build → canvas → refine; metric discovery; dashboard layout; chart/component authoring; workspace controls
Method
Static code trace, requirements/test comparison, production build, direct local browser/API verification. No application code changed.
Bottom line
The workbench is a convincing interaction prototype around a deterministic forecast demo, not yet a comprehensive agentic dashboard generator. The core gap is architectural: plans do not represent a metric set or component queries, generated data is synthetic, layouts are ordinal rather than grid-based, and neither chat nor canvas can create a new component.

[Heading1] 1. What is implemented today
[Heading2] Runtime flow
[ListParagraph] A user prompt is classified by deterministic keyword matching into descriptive, diagnostic, predictive, or prescriptive intent.
[ListParagraph] The planner attempts an exact governed metric-name match and emits a session spec containing one target metric, one grain, a date range, one output type, and feature candidates.
[ListParagraph] Approval creates/versions the session spec and starts the common pipeline DAG.
[ListParagraph] The pipeline creates gold/model artifacts, then registers a fixed visualization specification list.
[ListParagraph] Chart rows are generated by a deterministic seeded function; the run saves a fixed self-contained HTML artifact.
[ListParagraph] The workbench loads chart rows and permits limited edits to existing layout sections.
[Heading2] Useful substrate worth retaining
[ListParagraph] Clear ask → clarify → plan → approve interaction and visible build stages.
[ListParagraph] Semantic catalog, governance manifests, data-quality gates, lineage/UAS, model cards, audit logs, sharing, comments, and version history provide valuable foundations.
[ListParagraph] A section PATCH endpoint distinguishes layout and semantic edits and versions layout changes.
[ListParagraph] The canvas already supports rename, ordinal reorder, three mark types, a target overlay, basic device widths, comments, and inspector tabs.
[ListParagraph] The production client build succeeds.
[Heading2] Architectural discontinuity
The trusted substrate and the dashboard canvas are not connected by a durable dashboard specification. The session spec describes an analysis at a high level; the visualization store contains fixed summaries; the chart table contains one generic series; and layout_json describes only presentation order. No single object says which metrics, dimensions, filters, calculations, queries, encodings, and grid placements make up the dashboard.
[Heading1] 2. Detailed findings
F-01
Critical
Generated data is not connected to the selected metric or source
Every run calls a deterministic seeded generator that emits the same 90-day series beginning Jan 15, 2024. The metric, source, requested date range, grain, and warehouse result do not influence chart_data.
Impact: Dashboards can look plausible while answering a different question. This invalidates business accuracy.
F-02
Critical
The planning contract supports one target metric only
The planner returns target_metric as a scalar and validate_session_spec requires that scalar. A multi-metric ask is not decomposed into a metric inventory, dependencies, comparisons, or component requirements.
Impact: The requested comprehensive metric list/series cannot be represented, validated, or queried.
F-03
Critical
Dashboard composition is a fixed forecast template
dashboard_plan and viz_specs use hard-coded panel arrays. Workbench forces a 14-day horizon when a descriptive plan has no horizon, and the pipeline still trains/renders forecast content.
Impact: Descriptive, diagnostic, predictive, and prescriptive asks converge on essentially the same dashboard.
F-04
Critical
There is no grid layout model or drag/drop/resize implementation
layout_json stores only id, title, mark, top_n, and ordinal position. The canvas is a vertical flex column. The visible drag handle has no drag handlers and the live page contains zero draggable elements.
Impact: Users cannot compose a dashboard spatially or resize/reflow components.
F-05
Critical
No component creation or deletion workflow exists
The API only PATCHes known section IDs. There is no POST/DELETE section endpoint, component registry, chart builder, query builder, or blank-grid add flow. Chat always creates another session plan.
Impact: New charts, KPIs, tables, text, and controls cannot be added from chat or the workbench.
F-06
High
Freshly built workbench drops two persisted sections
save_artifact builds a four-section layout in the database after reading the artifact row, then returns the stale pre-layout row. BuildCanvas sees no layout_json and seeds only timeseries_ci and forecast.
Impact: The live dashboard omits dimension breakdown and feature importance even though the saved record contains them.
F-07
High
Canvas Export is wired to an unsupported format
The button opens /export?format=html, while the route accepts only csv or json. Direct verification returned HTTP 400.
Impact: A prominent implemented button is non-functional.
F-08
High
Canvas edits do not drive the exported artifact
Chart-type edits trigger a re-render, but generate_artifact_html does not consume layout_json and emits its fixed panel markup. Title/position changes do not re-render by design.
Impact: The editable canvas and the saved/shareable artifact diverge.
F-09
High
Post-build chat is not a true refinement loop
Follow-ups invoke the original planner and can start a new pipeline on the same session, but BuildCanvas retains the old artifact. Its save effect exits when artifact is already set, so the new run is not assembled into the current dashboard.
Impact: Prompts such as “add a promo overlay” do not patch the displayed dashboard.
F-10
High
Governance and trust indicators overstate what was validated
The GOVERNED and CONTRACT labels are rendered unconditionally in key canvas surfaces. Planning can proceed without a semantic schema or governance manifest, and the SQL VALIDATED badge is hard-coded green.
Impact: Trust language can be shown without the evidence it implies.
F-11
High
Per-component data/query contracts are incomplete
Contracts are persisted only for KPI row, timeseries, and forecast. Dimension breakdown and feature importance lack component queries/contracts, while all canvas charts consume the same generic chart_data rows.
Impact: The system cannot prove metric bindings, row shape, freshness, or query lineage for every component.
F-12
High
Workbench sessions are not resumable
Loading /app/create/:sessionId does not hydrate prior messages, confirmed specs, runs, artifacts, or layout. The route shows an empty approval placeholder until a new in-memory interaction occurs.
Impact: Refresh/deep-link loses the agentic workbench state.
F-13
Medium
Chart, filter, target, and responsive semantics are shallow
Only line/bar/area are live; scatter/treemap/table are disabled. The only filter is local “Hide forecast.” The target is a synthetic +5% average and is not persisted. Mobile only narrows the container while KPI columns remain fixed at four.
Impact: The editor cannot express common dashboard analysis patterns.
F-14
Medium
Several controls are misleading or unsafe
Skip to result only hides telemetry; Move can create out-of-range/gapped positions; version text is always v1; errors are broadly swallowed; the section PATCH route lacks the role/ACL protection used by other artifact mutations.
Impact: Controls can appear successful while state, safety, or error recovery is incorrect.
F-15
Medium
Regression tests validate parity slices, not end-to-end generation
Tests assert visible sections and rename/reorder, but not source fidelity, multiple metrics, add/delete, drag/resize, reload, exported-layout parity, refinement, or every control. Today the focused UI suite was also blocked at browser launch by a missing Chromium executable.
Impact: The completion ledger can remain green while product-defining behaviors are absent.

[Heading1] 5. Required target design
[Heading2] A. Durable dashboard specification
Introduce one canonical, versioned DashboardSpec shared by planning, execution, canvas, export, sharing, and presentation. It should be the source of truth rather than layout_json plus fixed renderer assumptions.
Area
Required fields
Metric inventory
metrics[]: canonical ID/version, label, role (primary/supporting/derived/target), expression/dependencies, format, aggregation, owner, confidence, access decision
Analysis
intent, questions, decisions, time range, grain, comparisons, forecast settings, assumptions
Dimensions/filters
dimensions[], globalFilters[], componentFilters[], allowed values, default state, cross-filter behavior
Components
components[]: id, type, title, metric refs, dimension refs, query spec, transform, encoding, interaction, empty/error state, contract ID
Grid
layouts per breakpoint with x, y, w, h, min/max sizes, lock state, z-order
Trust
source lineage, semantic versions, query hash, row count, freshness, DQ gates, access policy, generated-at
Lifecycle
spec version, parent version, patch history, author (agent/user), validation status, artifact/render versions
[Heading2] B. Agent planning process
[ListParagraph] Decompose the request into business questions and decision outcomes.
[ListParagraph] Resolve every named and implied metric against the semantic catalog; return a visible metric checklist with unresolved/ambiguous items.
[ListParagraph] Expand derived metrics into dependencies and identify targets/comparisons (for example revenue, target revenue, target gap %, order count, AOV, conversion).
[ListParagraph] Choose dimensions, time grains, filters, and comparison periods with confidence and access checks.
[ListParagraph] Propose components based on analytical role, not a fixed template.
[ListParagraph] Generate and validate a read-only query plan per component; preview row shape and cost.
[ListParagraph] Select visual encodings using data type/cardinality and user intent.
[ListParagraph] Pack components into responsive grid layouts and show the full component plan before build.
[ListParagraph] Execute, validate contracts, assemble from DashboardSpec, and preserve exact parity across canvas/export/share/present.

[Heading1] 6. Grid and authoring requirements
[Heading2] Grid behavior
[ListParagraph] Use a 12-column desktop grid with explicit x/y/w/h geometry; store tablet and mobile layouts separately or derive them deterministically and allow overrides.
[ListParagraph] Drag by a real handle, resize from visible edges/corners, show collision/placement previews, and normalize positions after every mutation.
[ListParagraph] Support undo/redo, keyboard movement/resizing, multi-select, duplicate, delete, lock, grouping, and accessible non-pointer alternatives.
[ListParagraph] Persist each operation as a versioned dashboard patch with optimistic concurrency and rollback.
[ListParagraph] Render the same DashboardSpec in workbench, artifact HTML, public viewer, exports, and present mode.
[Heading2] Chat authoring
After the first build, chat should switch from session planning to dashboard patch intent. The agent should generate a proposed patch, explain affected metrics/queries/layout, preview it, and apply only after confirmation when the change is material.
User request
Expected patch
“Add AOV and order count as KPIs”
Resolve two metrics, add two KPI components, create queries/contracts, repack grid
“Break the gap down by region”
Add or modify a bar component using target_gap_pct × region
“Turn this into a table”
Change component type and encoding while preserving metric/query semantics
“Put forecast below the KPIs and make it wider”
Layout-only grid patch; no data rerun
“Use weekly instead of daily”
Semantic patch; rerun affected queries/model and revalidate dependents
[Heading2] Workbench authoring
[ListParagraph] Add Component opens a palette: KPI, line, bar, area, scatter, table, heatmap, treemap, narrative, filter, image/spacer.
[ListParagraph] A component builder lets the user choose metrics, dimensions, aggregation, filters, comparison, sort/top-N, format, and interaction.
[ListParagraph] The builder shows a live data preview, query/contract status, and a chart recommendation without disabling alternatives silently.
[ListParagraph] Deleting or changing a component shows downstream impact and creates a reversible version.

[Heading2] Recommended sequencing notes
[ListParagraph] Do not start with a drag/drop library. First define DashboardSpec and server mutation semantics so grid operations have a durable target.
[ListParagraph] Do not extend artifact_gen with more fixed panels. Replace the fixed assembly path with component renderers driven by the same spec used by the workbench.
[ListParagraph] Keep semantic resolution deterministic and governed; an LLM may propose candidates, but canonical IDs, expressions, joins, access, and query validation must be server-verified.
[ListParagraph] Treat layout-only and semantic patches differently: layout changes should be instant; metric/dimension/grain changes should selectively rerun affected components.

[Heading1] 9. Final assessment
The project has unusually rich governance, lineage, model, artifact, and collaboration substrate. The gap is not a lack of surrounding platform capability; it is the missing middle between a business question and a durable, source-bound dashboard specification.
The fastest credible route is:
[ListParagraph] Make generated data real and metric/source-bound.
[ListParagraph] Represent all required metrics and components in a versioned DashboardSpec.
[ListParagraph] Build component create/delete/query/contract APIs.
[ListParagraph] Place those components on a persisted responsive grid.
[ListParagraph] Make chat and the workbench produce the same validated patch operations.
[ListParagraph] Render every delivery surface from that same specification and test every enabled control.
Until those steps are complete, the workbench should be described as a governed forecast-dashboard prototype rather than a comprehensive agentic dashboard generator.
