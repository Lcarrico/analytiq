# AnalytIQ UI Requirements Document — Full Mockup Analysis

Exhaustive breakdown of every HTML file in the "AnalytIQ UI Requirements Document" package: structure, layout, sections, intent, styling, components, and routes. Source: static high-fidelity HTML mockups (PRD v3.0), 34 `.dc.html` canvas files + hub + bundle, ~85 screens total.

---

## 1. Package format & rendering system

**File type.** Every screen file is a `.dc.html` "design canvas" document. Actual markup lives inside a custom `<x-dc>` element; a shared React-based runtime (`support.js`, generated from `dc-runtime/src/*.ts`, ~60KB) parses the `<x-dc>` template at load, mounts it with `window.React`/`window.ReactDOM`, and executes an optional per-file `<script data-dc-script>` (a `DCLogic` component class) for interactivity. Conditional regions use custom `<sc-if>` elements bound to component state.

**Head/helmet (identical on every file):**
- `<meta name="design_doc_mode" content="canvas">`
- Google Fonts preconnect + `IBM Plex Mono (400/500/600)` and `IBM Plex Sans (400/500/600/700)`
- Single style block: `body{margin:0;-webkit-font-smoothing:antialiased}` and `*{box-sizing:border-box}` — **all other styling is inline** on elements (no classes, no CSS framework)

**Canvas board convention.** Each file is a board, not a page:
- Board root: `min-height:100vh; background:#e9ebef; padding:48px` (56px on Index), `display:flex; flex-direction:column; gap:16px; align-items:flex-start; font-family:IBM Plex Sans`
- Board breadcrumb strip at top: mono 11px `#5b6478` — `AnalytIQ / App / <File area>` plus pill links (`border:1px solid #c7d9f8; border-radius:999px; padding:3px 10px; background:#eff4ff; color:#2563eb`) to `Index.dc.html` ("All screens ↗") and sibling files
- Each screen = a `<section id="…" data-screen-label="Screen Name /route">` containing a mono 11px frame label (`NN · Screen Name · /route [· content crop]`) above a **frame**: `background:#fff` (or `#f7f8fa` for full app shells), `border:1px solid #d8dce3`, `border-radius:12px`, `box-shadow:0 4px 20px rgba(15,23,42,.07)`, `overflow:hidden`. Full app screens are 1440px wide; content-crop frames 1180px; panels/modals 340–700px; Workbench 1500×900
- Files cross-link via `href="<File>.dc.html#<section-id>"` so the whole set browses like a clickable prototype

**Companion files:**
- `PLAN.md` — build plan & committed design language (checklist of all 34 files, all checked)
- `AnalytIQ UI - Screen Directory.html` (472KB) — self-extracting single-file bundle ("Bundled Page"): shows an SVG thumbnail + "Unpacking…" loader, then decodes a base64 manifest of the Index hub + assets into blob URLs. Distributable standalone version
- `export-src/Index Standalone.dc.html` + `export-src/support.js` — source pair used to produce that bundle
- `.thumbnail` — WebP board thumbnail

## 5. Cross-cutting observations

**Interactivity inventory.** Only Create Workbench ships real logic (DCLogic 5-state switcher via `<sc-if>`). Everything else is static but *navigationally* interactive: ~300 `href` cross-links make the 34 files browsable as one prototype. Anchors double as route stand-ins.

**Consistency mechanics.** The app shell (sidebar 240 + topbar 64) is copy-pasted verbatim into every 1440px frame with only the active nav item changed — 6 full-shell screens (Home, Activity, Library, Sources, Admin, Billing). Denser screens use 1180px "content crop" frames that omit the shell, and small panels/modals are rendered bare. Font, badge, table, and card recipes are byte-identical across files, confirming PLAN.md's committed design language was enforced.

**Recurring signature patterns worth carrying into implementation:**
- Trust surfaced everywhere: health scores 0–100, gate chips (6/6 ✓), CONTRACT PASSED, confidence values on every AI-derived object (0.58–0.99), GOVERNED badges, "read-only" reassurances, audit-log notes on destructive/approval actions.
- Admin/technical duality: business-friendly view by default with dark mono "technical detail · admin only" blocks (SQL, logs, contracts) behind toggles — mirrored by the Settings "Show technical detail" preference.
- Dark `#0b1220` is reserved for: marketing hero/footer/quote bands, code/SQL/log blocks, present mode, and the Enterprise plan card — it always signals "technical depth or theater."
- Numbers are always mono; every table is a CSS grid with a bespoke fr-template; every status is a tinted mono pill; anomaly rows get `#fdf9ef`/`#fdf6f6` washes.
- One continuous demo narrative (acme-retail, Q3 target risk, wms_events failure, discount_pct drift, zip4 PII) threads through all ~85 screens, so any screen can deep-link to any other and stay coherent.
